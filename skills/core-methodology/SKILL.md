# Core Methodology

## 适用场景 / 触发条件

- 任何新题目都可以先用它建立全局节奏。
- 对外部 Web、内网、CVE、社交型智能体题都适用。
- 尤其适合“信息多但真假未分”“步骤多但顺序不清”的局面。

## 何时加载

- 第一次看到题目时。
- 已经开始尝试，但仍说不清目标、入口、约束和当前已确认事实时。
- 连续两轮行动没有带来新事实时。

## 输入线索 / 识别信号

- 只给了 URL、描述或少量账号，没有现成攻击路径。
- 现场信息里混杂推测、希望和事实，边界不清。
- 多种技能似乎都能用，但没有证据说明哪条路收益最高。

## 主要目标

- 明确题目的胜利条件、边界条件、可操作入口和当前阻塞点。
- 把行动拆成“侦察 -> 验证 -> 利用 -> 取证/退出”的小闭环。
- 建立切换技能的标准，而不是靠感觉乱跳。

## 推荐工具

- `ListSkills` / `LoadSkill`
- `mcp__sandbox__execute_code`
- `toolset.note`
- 按场景补 `toolset.browser`、`toolset.proxy`、`toolset.terminal`

## 执行步骤

1. 把题目转成四个问题：目标是什么、入口在哪里、当前约束是什么、有哪些已确认事实。
2. 先确认边界：目标主机、域名、账号、端口、比赛规则、时间限制、不可用能力。
3. 把“猜测”与“事实”分开，只把事实记到 note 或 workspace 产物中。
4. 用最便宜的动作收集下一批事实，例如页面结构、响应头、错误回显、版本信息、登录流程、会话形态。
5. 每次只推进一个假设：例如“先确认是否有隐藏 API”，不要同时混做扫描、爆破、利用。
6. 一旦发现敏感事实，立刻记录：凭据、版本、内网地址、可疑参数、异常回显。
7. 当某条路径满足明确场景特征时，再加载对应技能。
8. 当获得旗标、关键口令或比赛判定成功条件后，立即结束无关探索。

## 关键检查点 / 决策点

- 现在的目标是“找入口”还是“证实漏洞”还是“拿结果”？
- 当前动作是否会产出新的事实，还是只是在重复尝试？
- 是否已经有足够信号表明应切换到 Web、CVE、OA、AD 或智能体社交技能？
- 是否存在更小、更快、更可观测的验证步骤？

## 失败时如何切换策略

- 如果当前路径两次都没有新事实，退回上一层，把问题改成更小的验证问题。
- 如果自动化工具没有收获，回到功能与流量本身，重新画目标逻辑图。
- 如果耗时增长但收益下降，切换到能提供更强结构化指导的场景技能。
- 如果会话不稳定，先转入 `persistence-maintenance`，再继续进攻。

## 需要记录的证据 / Notes

- 目标与胜利条件。
- 已知入口、账号、路径、版本、错误信息、接口、参数。
- 每次验证的结果与结论，不要只记命令，不记结论。
- 为什么切换到某项技能，触发依据是什么。

## 成功判据 / 退出条件

- 已形成清晰的下一步路径，并知道要加载哪项场景技能。
- 或者已经拿到题目要求的最终结果。
- 或者明确证明当前路径不可行，并为下一路径准备好了事实基础。

## 常见误区 / 风险提示

- 一开始就把所有技能、所有工具、所有想法同时拉进上下文。
- 没有先确认边界，就直接利用或盲扫。
- 记了大量过程，却没有沉淀“已确认事实”。
- 把超时当失败，实际上它更常是“动作过大、粒度不对”的信号。
