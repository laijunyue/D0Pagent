# Network OA Pivot

## 适用场景 / 触发条件

- 第三赛区中 Web 到内网、办公系统、代理链、跳板链、多层网络通路规划。
- 入口是 OA、统一门户、审批系统、邮件系统、文件系统、VPN 管理台等。
- 已拿到一个入口，但目标资源在更深层网络或更高权限区。

## 何时加载

- 页面、接口或报错中出现内网地址、代理配置、管理平面路径时。
- 已经具备弱口令、文件上传、模板注入、RCE、会话或 token。
- 需要明确“下一跳去哪、用什么身份、怎么保持路线”时。

## 输入线索 / 识别信号

- OA 品牌指纹、审批接口、组织架构、通讯录、文档中心、流程引擎、管理后台。
- 代码或配置中出现 `10.x`, `172.16.x`, `192.168.x`、内网域名、数据库地址、Redis 地址。
- 会话、cookie、JWT、LDAP、AD、VPN、网闸、堡垒机、WAF 后台等线索。

## 主要目标

- 从现有入口向更高价值内网资源推进，而不是停留在外层页面。
- 拆分“入口利用 -> 身份获取 -> 代理建立 -> 目标发现 -> 横向/纵向”的路线。
- 让每一步都有可回退、可记录、可继续利用的中间成果。

## 推荐工具

- `toolset.terminal`：建立代理、跑轻量扫描、验证服务连通性。
- `toolset.note`：记录跳点、凭据、路由和网络分层。
- `mcp__msf__session` / `mcp__msf__workspace`：当已有会话或需要稳定路由管理时辅助使用。
- `mcp__sandbox__execute_code`：编写小段脚本做 HTTP、SOCKS、接口验证。

## 执行步骤

1. 把现有入口分类：
   - Web 管理面
   - 文件上传 / 模板 / 插件入口
   - 弱口令或默认口令
   - 会话、token、cookie、SSO 凭据
2. 识别 OA 的真实价值点：
   - 组织架构、通讯录、审批流、单点登录、文件管理、报表、消息推送、系统设置
   - 这些功能往往带来账号、内网接口、文件、数据库或管理入口
3. 建立网络视图：
   - 当前落点在哪一层
   - 能访问哪些内网段、主机名、服务端口
   - 哪些资源必须通过代理或跳板才能到达
4. 规划路径时遵循“最短可验证链”：
   - 先确认一条下一跳是否可达
   - 再确认下一跳是否有高价值身份或服务
   - 再决定是否扩展更多横向点
5. 对常见 OA 场景优先检查：
   - 弱口令、默认口令、历史口令
   - 文件上传、模板注入、导出导入、插件、在线预览
   - 内网接口转发、管理后台、补丁缺失、调试接口
6. 一旦拿到凭据或会话，立即记录其来源、作用域、有效期和可访问范围。

## 关键检查点 / 决策点

- 当前入口最适合拿“代码执行”“凭据”还是“代理能力”？
- 现在是应该扩深一层，还是先固化现有入口和证据？
- 这条多步路径里，哪个节点失败后成本最高，是否先降低该风险？
- 是否已经出现更适合切换到 `persistence-maintenance` 或 `ad-internal-ops` 的信号？

## 失败时如何切换策略

- 如果外层入口利用不稳定，先固化会话或找更稳的凭据路径。
- 如果代理链复杂但无高价值目标，退回 OA 本身继续挖管理接口和身份。
- 如果已确认进入域环境，切换到 `ad-internal-ops`。
- 如果只是需要稳定保活，不再盲扩，先切到 `persistence-maintenance`。

## 需要记录的证据 / Notes

- 每个跳点的地址、协议、凭据、可达目标、失效条件。
- OA 系统版本、模块、账号体系、管理后台入口。
- 发现的内网主机、域名、数据库、共享、服务端口。
- 每一层路径的进入方式和退出方式。

## 成功判据 / 退出条件

- 已经建立一条可重放的多步攻击路径。
- 已获得更深层网络的高价值入口、凭据或会话。
- 或者明确证明当前 OA 路径收益不足，应该回退换入口。

## 常见误区 / 风险提示

- 只盯着 OA 首页，不看它背后的组织、文件、接口、SSO 价值。
- 一拿到 Webshell 就盲扫整个内网，没有先确认路由和目标价值。
- 只扩点不留记录，导致路线一断就完全丢失上下文。
- 不区分“可达”和“值得继续打”的目标。
