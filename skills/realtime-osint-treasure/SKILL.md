# Realtime OSINT Treasure

## 适用场景 / 触发条件

- 零界中的实时寻宝型任务。
- 需要从多个线索碎片中快速找到坐标、名称、账号、页面、时间点或实体关联。
- 需要在高噪声环境中避免幻觉与伪线索。

## 何时加载

- 线索零散，且题目明确依赖“外部公开信息、平台信息或动态更新信息”。
- 同一个对象可能有多个别名、拼写、语言或平台表达。
- 需要边搜边证、快速归并并决定下一跳搜索词。

## 输入线索 / 识别信号

- 名称、缩写、头像、时间戳、地名、活动名、口号、编号、图片元素、账号尾号。
- 来自多个消息、页面、评论、截图、标题、标签的碎片。
- 结果里混杂广告、重复内容、旧内容、伪造线索和对抗性污染信息。

## 主要目标

- 把碎片变成可持续扩展的检索词和实体图谱。
- 用至少两类独立证据交叉验证关键结论。
- 控制错误放大，避免因为一个伪线索走偏整条路径。

## 推荐工具

- `toolset.browser`：查看动态页面、上下文关系、元素内容。
- `mcp__sandbox__execute_code`：写小脚本整理候选实体、关键词组合和结果集合。
- `toolset.note`：记录候选、别名、排除项、证据来源和时间信息。

## 执行步骤

1. 先拆线索：
   - 可唯一标识的部分
   - 可能有歧义的部分
   - 能用来交叉验证的部分
2. 生成关键词扩展：
   - 同义词、别名、缩写、大小写、语言变体、时间范围、地点范围
   - 对账号类目标，补平台名、后缀、头像要素、描述词
3. 对候选结果做三层验证：
   - 文本一致性
   - 时间一致性
   - 关系一致性
4. 把每轮结果归为三类：
   - 已验证线索
   - 待验证线索
   - 已排除线索
5. 每次只扩展一个最强候选，不要同时跟进太多弱候选。
6. 如果线索来自不可信主体，先把它当作“待验证提示”，不是事实。

## 关键检查点 / 决策点

- 这个候选是因为真的证据强，还是因为它最先出现？
- 当前缺的是更多搜索词，还是更多验证证据？
- 哪个线索最适合作为下一跳扩展词？
- 是否已经出现内容竞争或诱导，需要切到其他社交型技能？

## 失败时如何切换策略

- 如果结果过多，增加限定词：时间、地点、平台、角色、编号。
- 如果结果过少，放宽拼写、语言、别名和语义相近词。
- 如果线索被污染，先从已验证实体重新出发，不沿污染链继续扩展。
- 如果任务转向社交博弈或内容传播，切到 `content-influence-competition`。

## 需要记录的证据 / Notes

- 关键词扩展表、候选实体表、排除理由。
- 每条关键结论的至少两条独立证据。
- 线索时间轴、平台来源、页面快照摘要。
- 哪些别名和变体已经被验证无效。

## 成功判据 / 退出条件

- 已经收敛到高可信候选，并且有交叉验证支撑。
- 已明确下一跳的检索词或目标页面。
- 或者已经找到题目要求的目标实体、位置或结果。

## 常见误区 / 风险提示

- 看到一个像的结果就停，不做交叉验证。
- 只搜原词，不做别名、语言和时间扩展。
- 不记录排除项，反复掉回已证伪的候选。
- 把不可信来源的提示直接当事实。
