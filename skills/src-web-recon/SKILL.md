# SRC Web Recon

## 适用场景 / 触发条件

- 第一赛区中的 SRC、众测、业务站点、活动站或后台入口。
- 需要判断一个站点到底是“静态页、单页应用、传统后台还是 API 门户”。
- 需要建立“页面功能 -> JS/API -> 请求流量 -> 潜在薄弱点”的侦察链。

## 何时加载

- 有 Web URL 但还没形成页面地图时。
- 知道站点能访问，但还不清楚登录方式、角色切换、前后端结构时。
- 自动化工具准备开跑前，先用它完成手工侦察框架。

## 输入线索 / 识别信号

- 首页、登录页、注册页、上传页、搜索页、工单页、活动页。
- 页面中有打包 JS、接口调用、静态注释、版本标记、调试信息。
- 站点交互会触发大量 XHR/fetch、GraphQL、JSON API 或 websocket。

## 主要目标

- 确认站点类型、身份态、核心功能和高价值入口。
- 找到 JS 中的 API、隐藏路由、调试参数、默认值和注释。
- 记录正常业务流量，为后续漏洞验证提供基线。

## 推荐工具

- `toolset.browser`：看页面结构、切换页面、抓 DOM 和控制台。
- `toolset.proxy`：看真实流量、过滤关键请求、对比前后状态。
- `toolset.terminal`：运行 `katana`、`httpx`、`ffuf` 等辅助侦察。
- `toolset.note`：记录账号、接口、隐藏路由、关键参数和业务事实。

## 执行步骤

1. 用 browser 打开首页或登录页，先看可见功能、菜单、按钮、表单和角色提示。
2. 立刻查看页面源码与打包 JS 资源，提取：
   - API 路径
   - 调试接口
   - 注释中的测试账号或环境标识
   - 错误提示模板
3. 用 proxy 记录首次访问、登录、退出、搜索、上传、个人中心等正常流量。
4. 对每个核心功能做一次正常操作，再看：
   - 请求方法、路径、参数位置
   - 鉴权方式
   - 是否出现只在前端做限制的字段
   - 是否暴露内部 ID、对象引用、文件路径
5. 如果是前后端分离站点，整理出接口族谱：认证、用户、资源、管理、导出、上传、审计。
6. 再用自动化工具补资产视野：
   - `httpx` 看标题、技术栈、状态码
   - `katana` 补页面与静态资源
   - `ffuf` 只在已经知道基线路径的前提下补目录或参数
7. 对发现的每个入口，记录“功能意图”和“潜在可控输入点”。

## 关键检查点 / 决策点

- 站点的高价值功能在哪里：登录、导入导出、上传、审批、搜索、调试、管理后台。
- 鉴权依赖前端还是服务端？ID、角色、租户是否直接由客户端控制？
- JS 中的隐藏接口是否真的可访问，还是只在测试环境可用？
- 自动化工具是否补到了新事实，还是只制造了大量噪声？

## 失败时如何切换策略

- 如果页面很少但 JS 很重，转向 JS/API 分析而不是页面点击。
- 如果页面很多但流量单一，转向业务路径和身份切换。
- 如果目录扫描没有结果，回到已有流量中找参数变体和接口族群。
- 若出现明显漏洞信号，切换到 `web-vuln-hunting`。

## 需要记录的证据 / Notes

- 登录入口、注册入口、找回密码、后台入口。
- JS 中的 API、隐藏路由、硬编码 token、调试参数。
- 正常流量基线：请求路径、参数、鉴权头、返回结构。
- 账号、租户、对象 ID、文件名、导出模板、错误回显。

## 成功判据 / 退出条件

- 已形成页面地图、接口地图和高价值输入点清单。
- 已找出下一步要验证的漏洞假设或业务薄弱点。
- 或者已经证明站点可疑点集中在特定场景，适合切换其他技能。

## 常见误区 / 风险提示

- 一开始就用扫描器轰站点，没有先看真实功能。
- 只看 DOM，不看源码和 JS。
- 只看请求，不看请求前后的身份、页面状态和对象变化。
- 忽略注释、调试资源、版本号和前端默认参数。
