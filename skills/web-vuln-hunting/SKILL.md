# Web Vulnerability Hunting

## 适用场景 / 触发条件

- 已锁定具体接口、参数、上传点、导出点、模板点或对象引用。
- 想按优先级排查 Web 常见漏洞，而不是随机试 payload。
- 需要判断 `ffuf`、`sqlmap`、`nuclei` 什么时候有价值，什么时候反而浪费时间。

## 何时加载

- `src-web-recon` 已经告诉你哪里值得打。
- 页面、流量、对象 ID、错误信息已经足够支撑具体假设。
- 需要把“可能有问题”推进成“已验证漏洞或已排除”。

## 输入线索 / 识别信号

- URL、Body、JSON、Header、Cookie、JWT、文件名、模板变量、导出参数。
- 500 报错、SQL 关键字、模板表达式、回显差异、状态码变化、响应时间变化。
- 业务动作中的对象切换、租户切换、审批流、导入导出、秒杀/并发操作。

## 主要目标

- 用最短路径确认高价值漏洞是否成立。
- 区分“输入点很多”和“真正值得打的点”。
- 建立漏洞优先顺序，避免在低收益点过度消耗时间。

## 推荐工具

- `toolset.proxy`：对比正常请求与异常请求。
- `toolset.browser`：重放真实业务动作、观察前端校验与后端结果差异。
- `toolset.terminal`：在明确目标后使用 `ffuf`、`sqlmap`、`nuclei`。
- `toolset.note`：先记事实，再继续利用。

## 执行步骤

1. 先分层看输入点：
   - 标识类：ID、UUID、租户、角色、订单号、文件路径
   - 内容类：搜索词、模板内容、富文本、文件内容
   - 行为类：审批、导入、导出、回调、重置密码、发券
2. 优先排查高收益问题：
   - 认证绕过 / 授权绕过 / IDOR
   - 文件上传 / 路径处理 / 模板渲染
   - SQLi / 命令执行 / SSTI / SSRF / XXE
   - 逻辑漏洞、并发竞争、状态机缺陷
3. 对每种漏洞先做最小化验证：
   - SQLi：先看报错差异、布尔差异、时间差异，再决定是否上 `sqlmap`
   - XSS：先找回显位置和渲染上下文，再构造最小 PoC
   - SSTI：先探模板引擎特征，再逐步确认可执行能力
   - SSRF：先看服务端是否发起请求，再判断是否能触云元数据或内网
   - 上传：先看路径、后缀、MIME、内容、二次处理和回显位置
   - IDOR：先做同角色横向，再做越权纵向，再看租户边界
4. 仅在下列情况使用自动化：
   - 你已经知道参数点和请求模板，用 `ffuf` 扩枚举值
   - 你已经确认有 SQL 注入迹象，用 `sqlmap` 提升效率
   - 你已经知道站点结构并想补常见已知漏洞，用 `nuclei`
5. 自动化工具一旦无新增事实，立刻停，回到手工验证。

## 关键检查点 / 决策点

- 这个点是否有真实业务价值，而不仅仅是“能回显”？
- 现在的验证是否能区分“服务端处理”和“前端假象”？
- 自动化是否建立在明确请求模板之上？
- 这个漏洞如果成立，下一步的利用路径是什么？

## 失败时如何切换策略

- 如果常见注入都不成立，转向逻辑漏洞和对象边界。
- 如果自动化误报多，缩回最小 PoC 手工确认。
- 如果只有时间差没有回显，改走盲注或侧信道。
- 如果上传点只能存文件不能执行，转向文件读取、覆盖、解析链或二次引用。

## 需要记录的证据 / Notes

- 具体参数、请求样本、异常回显、状态码差异、时间差异。
- 哪些 payload 被验证过，结论是什么。
- IDOR 的对象关系、越权边界、租户与角色映射。
- 上传链的文件名、落地路径、访问路径和解析结果。

## 成功判据 / 退出条件

- 已确认至少一个可继续利用的漏洞或逻辑缺陷。
- 或者已排除该入口的大类漏洞，并知道应切换到哪条业务路径。
- 已有足够事实支撑下一阶段利用或取旗。

## 常见误区 / 风险提示

- 看到参数就上大字典或重扫描。
- 没先理解业务，就把所有问题都按“注入题”处理。
- `sqlmap` 没跑出来就认定没有 SQLi。
- 忽略 race、审批流、导出、重置密码等业务型高收益点。
